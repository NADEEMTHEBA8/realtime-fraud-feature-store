# ADR-002: PySpark Locally, Not in Docker

## Status
Accepted

## Context
PySpark can run locally (in the host's Python environment) or inside a Docker container. The MacBook Air has 16GB RAM, with Docker already using ~4GB for 5+ containers.

## Decision
Run PySpark locally against Kafka and MinIO in Docker. Spark connects via localhost ports.

## Consequences
- **Positive:** Saves 4GB+ RAM, avoids swap thrashing, faster iteration
- **Negative:** Requires Java 17 installed on host, JARs managed outside Docker
- **Trade-off:** Standard dev pattern. In production, Spark runs on a dedicated cluster.

# ADR-003: Pre-Downloaded JARs, Not --packages

## Status
Accepted

## Context
PySpark's `--packages` flag downloads JARs from Maven Central at runtime. This can fail behind corporate proxies, in air-gapped environments, or due to Maven outages.

## Decision
Pre-download all 6 required JARs (Kafka + S3A connectors) into `streaming/spark/jars/` and reference them via `spark.jars` config.

## Consequences
- **Positive:** Deterministic builds, works offline, no Maven dependency at runtime
- **Negative:** JARs must be manually updated when upgrading Spark versions
- **Trade-off:** Reliability over convenience. JARs are gitignored (large binaries).

# ADR-004: PostgreSQL as Warehouse (Not BigQuery)

## Status
Accepted

## Context
BigQuery was the original plan (Indian fintechs use it heavily), but GCP account creation was blocked due to credit card/regional restrictions.

## Decision
Use PostgreSQL (already running in Docker) with dbt-postgres. SQL models are ANSI-standard and warehouse-portable.

## Consequences
- **Positive:** Zero cloud cost, instant setup, same SQL works on BigQuery
- **Negative:** Missing BigQuery-specific features (partitioned tables, clustering, BI Engine)
- **Trade-off:** Adapter swap to dbt-bigquery is a 10-minute config change — zero model rewrites.

# ADR-006: Redis for Feature Serving

## Status
Accepted

## Context
The fraud scoring API needs features in <10ms. PostgreSQL can serve point lookups in 5-20ms. Redis serves from memory in <1ms.

## Decision
Use Redis as the feature serving layer. A loader script materializes dbt gold tables into Redis. Features have 24-hour TTL.

## Consequences
- **Positive:** Sub-millisecond latency (measured: 0.998ms), supports batch MGET
- **Negative:** Additional infrastructure, data duplication (Postgres + Redis), TTL management
- **Trade-off:** The latency improvement (20x faster than Postgres) justifies the operational overhead for real-time fraud scoring.
