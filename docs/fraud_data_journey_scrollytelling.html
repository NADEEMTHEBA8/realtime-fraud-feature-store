Step 1: The Event

A User Swipes a Card

The journey begins when our mobile app or web client hits our backend. The data arrives as raw, unstructured JSON. At this exact moment, we have no idea if this is a legitimate purchase or a stolen credit card testing a merchant.

Payload.json
{
  "event_id": "evt_981723A",
  "user_id": "u_4591",
  "merchant_id": "m_102",
  "amount_usd": 899.99,
  "timestamp": "2026-06-26T10:45:12Z"
}
Interview Quick-Hit

Q: Why not just use a static CSV?
A: A continuous Python generator proves I can handle unbounded streaming data, which is mathematically and architecturally different from a finite batch of CSV rows.

Step 2: Ingestion Buffer

Securing the Event in Kafka

We can't write directly to a database, because if Black Friday traffic spikes, the database will crash. Instead, we instantly drop the JSON payload into an Apache Kafka topic. Kafka acts as an infinitely scalable "shock absorber". The data is now a distributed byte-stream safely written to disk.

Kafka Record Metadata
{
  // Kafka wraps the payload with cluster routing info
  "topic": "transactions.live",
  "partition": 3,
  "offset": 981249,
  "key": "u_4591", // Partitioned by user
  "value": "{...raw payload bytes...}"
}
Interview Quick-Hit

Q: Why Kafka?
A: It acts as an infinitely scalable shock absorber. It prevents my database from crashing during traffic spikes and allows multiple downstream consumers to read the exact same event log without deleting it.

Step 3: Bronze Lakehouse

Spark Micro-Batching to S3

Apache Spark continuously reads the Kafka stream. Every few seconds, it groups thousands of JSON events together, enforces a schema, and converts them into highly-compressed, columnar Parquet files. It saves these to S3 using the Delta Lake format, creating our immutable "Bronze" layer.

Delta Log Entry
// Delta Lake ensures ACID transactions on S3
{
  "add": {
    "path": "part-0003-a2b.snappy.parquet",
    "size": 1048576,
    "partitionValues": {"date": "2026-06-26"},
    "stats": "{\"numRecords\": 5000, \"minValues\": ...}"
  }
}
Interview Quick-Hit

Q: What is a Lakehouse?
A: Delta Lake merges the cheap, massive scale of S3 object storage with the ACID transactions (updates/deletes) of a traditional database by utilizing Parquet files alongside a transaction log.

Step 4: The Data Warehouse

Relational Loading

S3 is great for cheap storage, but terrible for low-latency joins. We ingest the raw Delta files into a relational PostgreSQL Data Warehouse. Now, the transaction data sits next to our slowly changing dimensions (like User Profiles and Merchant Categories synced via Debezium CDC).

SQL View
-- Data is now strictly typed and queryable
SELECT 
  txn.user_id,
  txn.amount_usd,
  merch.category -- Joined from CDC stream!
FROM bronze_transactions txn
JOIN silver_merchants merch 
  ON txn.merchant_id = merch.id;
Interview Quick-Hit

Q: Why move the data to Postgres?
A: While S3 is great for raw scale, Postgres provides the ultra-low latency SQL joins and relational constraints needed by BI analysts and downstream dbt modeling.

Step 5: Silver & Gold

dbt Feature Engineering

Raw data doesn't catch fraud; behavior catches fraud. dbt (Data Build Tool) runs complex SQL aggregations on the warehouse data. It calculates historical behavior (e.g., "How much has this user spent in the last 24 hours?"). This highly refined dataset is our "Gold" layer.

Gold Feature Row
{
  // The output of our dbt aggregations
  "user_id": "u_4591",
  "24h_spend_velocity": 1250.00,
  "7d_txn_count": 45,
  "distinct_merchants_24h": 8
}
Interview Quick-Hit

Q: What problem does dbt solve?
A: It brings software engineering best practices to SQL. It handles dependency resolution, testing, and CI/CD, guaranteeing that the mathematical features feeding my ML models are never broken.

Step 6: The Serving Layer

Syncing to Redis

Postgres analytical queries are too slow (hundreds of ms) for a live checkout flow. We take the Gold features generated by dbt and push them into Redis, an incredibly fast in-memory key-value store. The data is now staged and ready for instant retrieval.

Redis Key-Value Pair
// Ultra-fast O(1) lookup structure
"HGETALL" "features:u_4591"
{
  "velocity": "1250.00",
  "count": "45",
  "merchants": "8"
}
Interview Quick-Hit

Q: Why add Redis to the stack?
A: Postgres is too slow for a <50ms checkout flow. We pre-compute features via dbt and push them to Redis for instant O(1) in-memory retrieval by the live application.

Step 7: Live Inference

The ML Model Strikes

The loop closes. When the next transaction arrives at the FastAPI endpoint, the API instantly pulls the historical features from Redis, combines them with the live transaction, and hands the unified array to the Machine Learning model. A fraud decision is returned to the user.

Latency: 42ms
// The final API Response
{
  "transaction_id": "evt_981723A",
  "status": "BLOCKED",
  "fraud_probability": 0.96,
  "reason": "High velocity + New Merchant Category"
}
Interview Quick-Hit

Q: How does the model stay fast?
A: Because 99% of the heavy lifting (aggregations, joins) is done asynchronously overnight. The live API only has to execute a simple matrix multiplication on pre-calculated numbers.
